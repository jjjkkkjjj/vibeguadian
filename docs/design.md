## 1. システム全体アーキテクチャ

ツールは単一のRustバイナリ（`vs` コマンド）として動作します。開発者は `vs run -- npm run dev` のように、普段のコマンドをラップして実行します。

### 技術スタック（推奨Crates）
* **CLIパーサー:** `clap`（サブコマンドや引数解析）
* **非同期ランタイム:** `tokio`（プロキシサーバーとプロセス管理の並行処理）
* **HTTPプロキシ:** `axum` または `hyper` ＋ `reqwest`
* **文字列検索（ログマスク）:** `aho-corasick`（複数のシークレットを高速に同時検索・置換）
* **シリアル化:** `serde`, `serde_json`

## 2. コア機能の詳細設計

### 機能A: Inject Mode（環境変数のメモリ注入）

プロジェクト外に保存されたシークレットを読み込み、子プロセス（開発対象のアプリ）のメモリ上にだけ展開して起動します。

* **設計詳細:**
    1.  `~/.vibesafe/secrets.json`（またはOSのキーチェーン）からシークレット群をパース。
    2.  Rustの `std::process::Command` を使用して子プロセス（例: `npm`）を構築。
    3.  `Command::envs()` を使って、パースしたシークレットを環境変数としてセット。
    4.  現在のシェルから引き継ぐ環境変数とマージしつつ、**ディスク（`.env`）には一切書き出さずに**子プロセスを `spawn()` します。

### 機能B: Proxy Mode（ローカルAPIプロキシ）

アプリが直接外部APIを叩くのを防ぐため、Rust側で軽量なリバースプロキシを立ち上げ、背後でヘッダーにAPIキーを注入します。

* **設計詳細:**
    1.  `vs run` 実行時、子プロセスを立ち上げると同時に `tokio::spawn` でバックグラウンドのローカルサーバー（例: `localhost:8080`）を起動。
    2.  **ルーティングマップの解決:** プロジェクト内の `vibesafe.toml` などから「どのパスへのリクエストを、どこへ転送し、どのキーを付与するか」の設定を読み込む。
    3.  **リクエストの傍受と転送:**
        * アプリからのリクエスト: `POST http://localhost:8080/proxy/stripe/v1/charges`
        * Rustプロキシが書き換え: `POST https://api.stripe.com/v1/charges`
        * ヘッダー追加: `Authorization: Bearer sk_live_...`
    4.  レスポンスをそのままアプリへ返す。

### 機能C: Log Mask Mode（ターミナル出力の検閲）

子プロセスが吐き出す標準出力（stdout）と標準エラー出力（stderr）を横取りし、シークレット文字列が含まれていないか監視・置換してからターミナルに表示します。

* **設計詳細:**
    1.  `Command` 実行時、`Stdio::piped()` を指定して子プロセスの stdout/stderr をRust側で捕捉。
    2.  `tokio::io::BufReader` で出力を1行ずつ非同期に読み込む。
    3.  **高速マスク処理（重要）:**
        * 一般的な正規表現（Regex）で複数のキーを毎行チェックすると、ログが大量に出た際にCPUを食い潰し、ボトルネックになります。
        * ここで **Aho-Corasick法**（Rustの `aho-corasick` クレート）を使用します。複数のシークレット文字列を事前コンパイルし、$O(n)$ の線形時間で一括置換（`***[MASKED]***` に変更）できるため、ログのストリーミングに対して圧倒的なパフォーマンスを発揮します。
    4.  マスク済みの文字列を、開発者のターミナル（親プロセスの stdout）へ `println!` で出力。

## 3. ディレクトリとデータ構造

AIエージェントに読み取らせないため、設定と実データの保存場所を明確に分離します。

| ファイル | 役割 | AIのアクセス |
| :--- | :--- | :--- |
| `~/.vibesafe/secrets.json` | 実際のAPIキーやDB接続文字列を保存（将来的には暗号化）。 | **不可**（プロジェクト外） |
| `./vibesafe.toml` | プロジェクト直下に置く。プロキシの転送ルールや、どの環境変数を注入するかの「名前の定義」だけを記載。 | **可能**（キーの本体はないので安全） |

> **セキュリティの要:** AIは `./vibesafe.toml` を読み取って「このプロジェクトはStripeとSupabaseを使っているな。エンドポイントは localhost:8080/proxy/... だな」と理解してコードを書けますが、本物のキーには物理的に触れられません。
